# ECL — Engineering Change Lifecycle

This is the operating manual for how AI agents manage changes in the Drill Practice Platform repository.

## 1 Purpose And Scope

ECL defines when and how changes are tracked, reviewed, verified, and archived. It applies to all AI agents working on this repository. The goal is explicit change state, reproducible verification, and learning from archived evidence.

## 2 When To Create A Change

Create an ECL change (`harness/changes/active/`) when any of these apply:

- The work spans more than two files.
- The work touches UI architecture, task data format, test runner, i18n structure, or cross-module behavior.
- The work needs user confirmation or plan review.
- The work needs multi-step tests or regression checks.
- The work is likely to take more than 20 minutes.
- A failure needs to be captured as a new constraint, task, or regression.

**Small, local fixes can skip `harness/changes`** if the final response records validation. A small fix is one file, low risk, no data/permission/architectural impact (e.g., fixing a typo in README, adjusting a CSS value, updating a single translation string).

Decision tree:
1. Active change exists → keep using it.
2. Small Change → copy, comments, README, formatting, single-file bug fix with no runtime impact.
3. Structured Change → APIs, data model, architecture, multiple files, unclear requirements, > 20 min.
4. Unclear impact → inspect first. If uncertainty remains, upgrade to Structured Change.

## 3 Change Lifecycle

```
new → active (in_progress)
active → close completed → archive/YYYY-MM-DD-slug
active → close blocked → archive/YYYY-MM-DD-slug
active → close abandoned → archive/YYYY-MM-DD-slug
active → park → parking/YYYY-MM-DD-slug
parking/slug → resume → active
```

**Rules:**
- Only one active change at a time.
- Starting a new task must never overwrite an existing active change.
- `harness/changes/INDEX.json` is generated by script only — never hand-edit.
- `docs/STATUS.md` is a handoff summary; active change files override it.

## 4 Change Template

Every active change uses 5 files:

```
harness/changes/active/
  summary.md    — Front matter (title, status, phase, reviews) + outcome, decisions, validation
  spec.md       — WHAT/WHY: goal, acceptance criteria, non-goals, constraints
  plan.md       — HOW: technical approach, impacted files, risks, verification plan
  tasks.md      — Executable tasks with T### ids
  reviews/
    review.md   — Intake, spec, plan, code, validation review statuses
```

## 5 Plan Review Gate

Before `phase: "implement"`, `plan_review` must be `"approved"` in `summary.md` front matter or an equivalent "Plan Review: approved" entry in `reviews/review.md`.

`spec.md` must have no unresolved `[NEEDS CLARIFICATION: ...]` markers before implementation.

## 6 Context Loading Order

1. `AGENTS.md`
2. `docs/ECL.md`
3. If active change exists: `harness/changes/active/summary.md` → `spec.md` → `plan.md` → `tasks.md` → `reviews/`
4. If no active change and `harness/evolution/pending.md` exists: read it before `docs/STATUS.md`
5. If no active change and no pending evolution: `docs/STATUS.md`
6. Task-specific docs from AGENTS.md section 3
7. `harness/changes/INDEX.json` only when selecting historical context
8. Selected archived `summary.md` files only

**Never load all archive history.** Use STATUS paths or INDEX search.

## 7 STATUS Handoff Protocol

Before closing active work:
1. Update `docs/STATUS.md` from active change summary/spec/plan/tasks/reviews.
2. Run `node scripts/harness-change.mjs close <completed|blocked|abandoned>`.
3. After close, update `docs/STATUS.md` with the final archive summary path.
4. Run `npm run lint:harness` to confirm ECL integrity.

## 8 Auto-Evolve

Core ECL includes threshold-based auto-evolve:

- `harness-change close` and `reindex` trigger `harness-evolve check`.
- When enough archived changes accumulate (default: 5), `harness/evolution/pending.md` is generated.
- `pending.md` is a maintenance reminder, not a hard lock. Reading it does NOT start auto-evolve.
- When no active change exists and pending maintenance is present, the agent should ask the user whether to handle it now.
- Auto-evolve flow: proposal → independent review → apply → verify → `mark-complete`.
- Generated scripts only perform mechanical counting; agents do semantic work.

## 9 Script Command Reference

All commands use Node.js scripts via npm or direct `node` execution:

```bash
# Change management
node scripts/harness-change.mjs new <title>
node scripts/harness-change.mjs status
node scripts/harness-change.mjs validate
node scripts/harness-change.mjs park [reason]
node scripts/harness-change.mjs resume <id>
node scripts/harness-change.mjs close <completed|blocked|abandoned>
node scripts/harness-change.mjs search <query>
node scripts/harness-change.mjs context
node scripts/harness-change.mjs reindex

# Auto-evolve
node scripts/harness-evolve.mjs check
node scripts/harness-evolve.mjs collect
node scripts/harness-evolve.mjs mark-complete

# Linters
node scripts/lint-ecl.mjs          # ECL structure integrity
node scripts/lint-encoding.mjs     # UTF-8 / mojibake check
node scripts/lint-deps.mjs         # Layer dependency enforcement

# Convenience
npm run lint:harness   # All harness checks
npm run lint:arch      # Architecture linters
npm run verify-harness # Full harness verification
```

## 10 INDEX.json Rule

`harness/changes/INDEX.json` is a generated search index. It is rebuilt by `park`, `close`, `resume`, and `reindex` commands. **Never hand-edit this file.**
